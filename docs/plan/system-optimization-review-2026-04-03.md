## 2. 已识别的当前优化机会（基于代码现状）

## O1. 认证入口一致性（高优先）

### 现状
- `/auth/login` 在 `wechat_code` 场景此前返回 501（未实现），而 `/auth/wx-login` 已实现 WeChat 登录流程。
- 这会导致同一业务语义存在双入口、行为不一致，增加客户端接入复杂度。

### 优化动作
- 将 `/auth/login` 的 `wechat_code` 分支接入 WeChat 登录逻辑。
- 抽取统一私有函数，复用给 `/auth/login` 与 `/auth/wx-login`，避免重复代码与行为漂移。

### 风险
- 低到中：影响 WeChat 登录路径，但不影响邮箱密码登录。

### 回滚
- 回退到上一个 commit，即可恢复“`/auth/login` 对 wechat_code 返回 501”的旧行为。

### 验收
- `POST /auth/login` + `wechat_code` 不再返回 501。
- `POST /auth/wx-login` 与 `POST /auth/login`（wechat_code 场景）响应结构一致。

---

## O2. 限流策略透明化（高优先）

### 现状
- 限流逻辑已实现（全局、公开端点、用户级），但当前缺少可观测字段（如限流命中来源、键类型）。

### 优化动作
- 为限流拒绝响应补充可观测标识（例如错误码或响应头），并统一日志字段。

### 风险
- 低：仅增强可观测性。

### 验收
- 能在日志中区分 global/public/user 三类限流命中。

---

## O3. Admin API 服务层“真实接口化”分阶段迁移（中优先）

### 现状
- 管理端大量接口仍采用 mock 数据，线上行为与开发行为可能出现差异。

### 优化动作
- 按页面分批把 mock 调用迁移到真实 API（先 Dashboard、再订单与审核）。

### 风险
- 中：涉及前后端联调与权限。

### 验收
- 每次仅迁移一个页面的数据源；页面可用性与错误提示保持一致。

---

## O4. 查询接口性能治理（中优先）

### 现状
- 已有分页，但尚未见到统一的慢查询画像与优化基线。

### 优化动作
- 先做慢接口清单（P95/P99），再做定点优化（索引、查询裁剪、序列化开销控制）。

### 风险
- 中：涉及数据库变更与兼容性。

### 验收
- 指定接口 P95 至少下降 20%。

---

## 3. 建议 PR 切分（下一步）

1. PR-1（已落地）：认证入口一致性（O1）
2. PR-2：限流可观测性增强（O2）
3. PR-3：Admin Dashboard 从 mock 到 real API（O3 第一阶段）
4. PR-4：热点查询基线采集与首批优化（O4）

---

## 4. 本轮结论

当前系统最值得优先推进的不是“大重构”，而是：
- **先消除行为不一致（认证入口）**
- **再增强观测能力（限流与错误码）**
- **最后做数据层性能优化（有基线再调优）**

这样可以最小风险地持续提升稳定性与交付效率。
