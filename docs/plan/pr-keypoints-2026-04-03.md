# VICOO 项目扩展与优化 PR 关键点（2026-04-03）

> 目标：在不牺牲稳定性和合规性的前提下，逐步推进功能扩展与工程优化。

## 1) 当前项目审查结论（简版）

### 1.1 架构层面
- 项目为多端架构（`backend` + `frontend/web-react` + `admin`），基础分层清晰，适合采用**小步快跑、纵向切片**的 PR 策略。
- 后端已具备统一中间件链路（CORS / 限流 / 安全头 / 统一异常）与模块化路由，说明可在现有基础上做“局部增强”而非大规模重写。

### 1.2 风险层面
- 认证路由仍存在显式待办：`WeChat login implementation`，属于高影响功能缺口，应避免与无关优化混入同一 PR。
- 配置项较多且安全/支付相关（JWT、支付、OAuth、邮箱、加密），任何“拓展 PR”都必须附带环境变量兼容策略与回滚方案。

---

## 2) PR 设计原则（务必执行）

1. **单目标 PR**：每个 PR 只做一个可验证目标（例如“登录流程补齐”或“查询接口性能优化”）。
2. **先观测后优化**：先补指标与日志，再做性能改造；禁止“盲改快”。
3. **兼容优先**：对外 API 的字段与语义尽量向后兼容，新增字段优于重命名字段。
4. **合规默认开启**：涉及儿童信息、支付、隐私的变更必须附最小合规检查清单。
5. **可回滚**：所有高风险改动必须具备 Feature Flag 或降级路径。

---

## 3) 推荐的 PR 分批路线（谨慎扩展）

## 批次 A：稳定性与可观测性（低风险，高收益）

### A1. 错误与性能可见化
- 在关键 API（auth、orders、payments、donations）补齐结构化日志字段：`trace_id`、`user_id`、`route`、`latency_ms`。
- 建立统一错误码对照表（业务错误 vs 基础设施错误），避免前端只能依赖 message 文案。

### A2. 限流与失败降级校准
- 对 `/health` 之外的公开接口分层限流（匿名/登录/管理端），并在异常时明确降级行为。
- 明确 Redis 不可用时的策略：Fail-open 还是 Fail-closed，并写入文档。

**验收标准**
- 提交后可通过日志检索定位 95%+ API 异常。
- 压测下服务在 Redis 短时不可用场景可预期退化，不出现雪崩。

## 批次 B：身份与账户链路补齐（中风险，高业务价值）

### B1. 完成 WeChat 登录待办
- 以独立 PR 实现并联调 `auth` 中的 WeChat 登录待办逻辑。
- 明确账号绑定策略：首次登录建号、已登录绑定、冲突账号处理。

### B2. OAuth/本地登录一致性
- 对 GitHub / Google / 密码登录统一用户资料落库规则，减少“同邮箱多身份”问题。

**验收标准**
- 登录后用户主键稳定、幂等。
- 失败分支（授权过期、state 不匹配、邮箱冲突）有可测试错误码。

## 批次 C：查询性能与数据一致性（中风险）

### C1. 热点接口 profiling
- 先识别 Top N 慢接口（campaign、artwork、order 列表）。
- 仅对确认慢查询添加索引或调整查询，避免“全库加索引”。

### C2. 分页与排序规范化
- 前后端统一分页参数与默认排序，避免管理端/用户端分页行为不一致。

**验收标准**
- P95 延迟下降（建议目标：20%+）。
- 分页接口在边界参数下（空页/超大页）返回一致。

## 批次 D：前端体验和发布质量（低~中风险）

### D1. Admin 与 Web 共性能力对齐
- 对管理端和 Web 端 API 调用层统一超时、重试、错误 toast 规范。

### D2. 构建与质量门禁
- 在 CI 中明确最小门禁：TypeScript 构建、后端测试、基础安全扫描。

**验收标准**
- 两端错误交互一致，用户可理解。
- 所有 PR 合并前必须通过统一门禁。

## 4) 每个“扩展/优化 PR”必须包含的内容模板

1. **变更目标**（一句话）
2. **影响范围**（backend/frontend/admin + 具体模块）
3. **风险分级**（低/中/高）
4. **回滚方案**（开关、回退版本、数据回滚）
5. **测试证据**（命令 + 结果 + 关键截图/日志）
6. **合规检查**（是否涉及儿童信息、支付、隐私）
7. **上线观察项**（错误率、P95、关键业务转化）

---

## 5) 不建议的做法（避免“看似优化，实则增险”）

- 单个 PR 同时改动：认证 + 支付 + 数据模型。
- 未给出基线数据就宣称“性能优化”。
- 修改配置默认值但不提供迁移说明。
- 在生产路径引入新依赖而不做容量评估。

---

## 6) 下一步执行建议（可直接开工）

- 第 1 个 PR：仅做“可观测性增强 + 错误码对齐”（批次 A1）。
- 第 2 个 PR：仅做“WeChat 登录待办补齐”（批次 B1）。
- 第 3 个 PR：仅做“热点慢接口优化”（批次 C1）。

这样可以把风险拆散，把收益前置，并让每个 PR 都可以独立评审、独立回滚。
