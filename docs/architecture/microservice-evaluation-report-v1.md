# 微服务拆分评估报告 (v1.0)

**日期**: 2026-04-02  
**项目**: VICOO-esp (童画公益 × 可持续时尚)

## 1. 背景与现状

当前项目采用 **模块化单体 (Modular Monolith)** 架构。虽然代码库在逻辑上划分为 `auth`, `user`, `artwork`, `donation`, `payment`, `supply_chain` 等模块，但它们：
- 共享同一个 FastAPI 进程。
- 共享同一个 MySQL 数据库和 Redis 实例。
- 具有较强的内存中调用依赖。

## 2. 拆分动力分析 (Drivers for Split)

### 2.1 支付模块 (Payment Service)
- **原因**: 支付属于高敏感、高一致性要求的模块。它需要处理多种第三方回调 (WeChat, Alipay, Stripe)，且容易受到外部网络抖动影响。
- **收益**: 独立部署可以隔离风险，支付服务的崩溃不会导致整个平台（如作品展示）不可用。可以实施更严格的 IP 白名单和安全审计。

### 2.2 捐赠模块 (Donation Service)
- **原因**: 捐赠逻辑涉及资金统计、证书生成（IO密集型）和审计记录。
- **收益**: 方便进行独立的数据对账和合规性审计。

### 2.3 AI 助手模块 (AI Assistant Service)
- **原因**: AI 调用（如 OpenAI）是高延迟的，且未来可能涉及本地模型推理，对计算资源（GPU/Memory）有特殊要求。
- **收益**: 资源隔离，防止长连接 AI 请求耗尽 API Gateway 的 worker 线程。

## 3. 拆分阻力分析 (Barriers to Split)

- **事务一致性**: 捐赠与支付目前在单体中可以通过本地数据库事务保证。拆分后需引入分布式事务（如 Saga 模式）或基于消息队列的最终一致性，开发成本显著增加。
- **运维成本**: 从 1 个容器增加到 8+ 个容器，涉及服务发现、链路追踪 (Tracing) 和复杂的 CI/CD 流水线。
- **团队规模**: 目前处于开发早期，单体架构的开发效率远高于微服务。

## 4. 最终评估结论

**建议阶段：维持模块化单体，但保持“服务化”代码风格。**

### 具体理由：
1. **流量规模**: 当前流量尚未达到单机 FastAPI 处理瓶颈。
2. **数据一致性**: 支付回调与捐赠状态更新的强一致性在单体中更容易维护，且不容易丢单。
3. **架构就绪**: 现有的 `Service` 层设计已经实现了良好的解耦。未来只需将 `Service` 包装成独立的 FastAPI 实例并修改数据库连接即可实现快速拆分。

## 5. 建议触发拆分的条件 (Triggers)

- 当 **支付模块** 需要通过 PCI-DSS 级别的高级安全认证时。
- 当 **作品展示 (Artwork)** 流量激增，需要独立进行大规模水平扩展以应对高并发投票时。
- 当 **AI 服务** 引入 GPU 推理，导致基础镜像体积超过 2GB 或有特殊硬件依赖时。
